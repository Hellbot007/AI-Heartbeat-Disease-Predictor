[7]: An Integrated ML + AI Heart Disease Risk Prediction System
[8]: 1. Aim
[9]: To study, design, and develop a full-stack, AI-powered heart disease risk prediction system that analyses patient physiological features and accurately classifies the likelihood of cardiovascular disease. The system integrates classical Machine Learning models — Random Forest, Gradient Boosting, K-Nearest Neighbors, and Logistic Regression — trained on a combined dataset of over 70,000 patient records and evaluated for accuracy and F1-score. An AI-driven chatbot powered by Google Gemini 2.5 Flash, combined with a Retrieval-Augmented Generation (RAG) engine and a document-parsing module, provides intelligent, grounded medical assistance beyond pure prediction.
[10]: 2. Project Objectives
[11]: To understand the clinical relevance and computational challenges of early heart disease detection using patient health metrics.
[12]: To collect, merge, and pre-process four independent cardiovascular and health datasets (Cleveland Heart Disease, Heart Failure Clinical Records, Cardiovascular Disease Dataset, and Sleep Health Dataset) into a single unified training corpus.
[13]: To implement data cleaning pipelines including type coercion, missing value imputation via mean-fill, and StandardScaler normalization.
[14]: To train and compare four ML classification models — Random Forest (primary), Gradient Boosting, K-Nearest Neighbours, and Logistic Regression — on 13 key physiological features.
[15]: To evaluate all models on a 20% held-out test set using Accuracy, F1-Score (Macro), and F1-Score (Weighted) as performance metrics.
[16]: To integrate the Google Gemini 2.5 Flash API for generating the top 3 most likely heart-related disease diagnoses with estimated risk percentages, and for conversational medical Q&A.
[17]: To implement a custom Retrieval-Augmented Generation (RAG) engine using TF-IDF vectorization and cosine similarity to ground AI responses in curated medical knowledge, reducing hallucination risk.
[18]: To build a multimodal medical document analysis feature allowing users to upload PDFs or images of lab reports, from which relevant health values are automatically extracted and fed into the prediction engine.
[19]: To design and deliver a premium, dynamic, landscape-oriented web UI using vanilla HTML, CSS, and JavaScript served via a Flask REST API backend.
[20]: 3. Problems Solved and How the System Solves Them
[21]: 3.1 Problem: Late Detection of Cardiovascular Risk
[22]: Heart disease is the leading global cause of mortality. Most patients receive a diagnosis only after a major event like a heart attack, by which point preventive intervention is too late.
[23]: How it is solved: The system provides an instant, data-driven risk assessment based on 13 routinely measurable physiological parameters (age, sex, chest pain type, blood pressure, cholesterol, etc.). Patients and clinicians can evaluate risk at any point without requiring specialist appointments, enabling timely intervention.
[24]: 3.2 Problem: High Dimensionality and Feature Noise in Clinical Data
[25]: Raw patient records contain dozens of measurements, many of which may be irrelevant, missing, or noisy. Feeding such data directly to a model without preprocessing leads to poor generalization.
[26]: How it is solved: A dedicated feature pipeline (feature_pipeline.py) selects exactly 13 clinically validated features. Numeric conversion and mean-fill imputation handle missing values. StandardScaler normalization ensures all features contribute proportionally to the model's decision boundary, preventing high-magnitude features (e.g., cholesterol in mg/dL) from dominating low-magnitude ones (e.g., binary fasting blood sugar).
[27]: 3.3 Problem: Single-Model Brittleness and Bias
[28]: Any single ML algorithm carries its own biases and failure modes. A single model may overfit to training data or perform poorly on certain patient demographics.
[29]: How it is solved: Four distinct ML algorithms (Random Forest, Gradient Boosting, KNN, Logistic Regression) are each independently trained and evaluated on the same 80/20 data split. Their comparative performance is analysed across Accuracy and F1-Score, allowing informed model selection. The primary production model (Random Forest) achieves 99.91% test accuracy.
[30]: 3.4 Problem: Raw Text Queries Cannot Be Matched to Medical Knowledge
[31]: An AI language model responding to cardiology questions from memory alone risks generating incorrect or hallucinated medical advice.
[32]: How it is solved: A custom RAG engine (rag_engine.py) converts both user queries and pre-loaded medical knowledge base entries into TF-IDF vectors. Cosine similarity retrieves the top-k most relevant knowledge snippets, which are injected into the Gemini prompt as verified medical context. This ensures AI responses are grounded in factual, curated cardiology knowledge.
[33]: 3.5 Problem: Patients Cannot Easily Translate Lab Reports into Predictions
[34]: Medical lab reports contain structured data (cholesterol levels, ECG readings, blood pressure) buried inside scanned PDFs or image files, making manual extraction time-consuming and error-prone.
[35]: How it is solved: The /upload-document API endpoint accepts PDF and image uploads and passes them to Google Gemini's multimodal vision capability, which extracts the relevant health values and automatically populates the prediction engine's feature inputs. This bridges the gap between raw medical documentation and the prediction interface.
[36]: 3.6 Problem: Heavy LLM Compute Requirements on Local Hardware
[37]: Running open-source LLMs locally (e.g., Ollama, LLaMA) demands high-end GPU infrastructure, making deployment impractical on standard servers or developer machines.
[38]: How it is solved: The system offloads all generative AI tasks (disease estimation, chatbot, document analysis) to the Google Gemini API. The ML prediction engine (Random Forest) runs in milliseconds on CPU using scikit-learn, while Gemini handles the computationally intensive reasoning tasks remotely. This makes the application deployable on any standard machine with internet access.
[39]: 4. System Overview
[40]: The AI Heartbeat Disease Predictor is a full-stack, hybrid AI web application that integrates classical machine learning with large language model capabilities for comprehensive cardiovascular risk assessment. The system accepts 13 patient physiological inputs through a browser-based interface, routes them through a trained ML classifier, and generates both a binary risk prediction (disease / no disease) and a ranked list of the top 3 most likely cardiac conditions with probability estimates. A conversational AI chatbot powered by Google Gemini 2.5 Flash and a medical document upload feature further extend the application's clinical utility.

[53]: 5. Mathematical Formulas Used
[54]: 5.1 Feature Standardization — StandardScaler (Z-Score Normalization)
[55]: Before training, all 13 input features are normalized using Z-score standardization. This ensures that features measured on different scales (e.g., cholesterol in hundreds vs. binary flags of 0/1) contribute equally to the model's learning process:
[56]: z = (x - μ) / σ
[57]: Where: x = original feature value,  μ = feature mean (computed on training set),  σ = feature standard deviation. Features with large raw values such as cholesterol (chol, ~200 mg/dL) would otherwise dominate distance-based algorithms like KNN without this normalization step.
[58]: 5.2 Mean Imputation for Missing Values
[59]: Several feature columns in the combined dataset contain missing values due to inconsistent recording across the source datasets. Missing values are filled using the column mean computed on the training data to preserve statistical properties:
[60]: x_imputed = mean(x_observed)
[61]: 5.3 Random Forest — Ensemble Decision (Majority Voting)
[62]: The primary model (Random Forest) aggregates predictions from N individual decision trees. Each tree T_i independently predicts a class label, and the final class is determined by majority vote:
[63]: ŷ = mode({ T₁(x), T₂(x), ..., T_N(x) })   N = 100 estimators
[64]: Each tree is trained on a random bootstrap sample (bagging) and uses a random subset of features at each split node, reducing overfitting and variance compared to a single decision tree.
[65]: 5.4 Gini Impurity — Decision Tree Node Splitting Criterion
[66]: Within each decision tree, node splits are chosen to minimize Gini impurity — a measure of how often a randomly chosen element would be incorrectly classified:
[67]: Gini(t) = 1 − Σ p(c|t)²   summed over all classes c at node t
[68]: A Gini impurity of 0 indicates a perfectly pure node (all samples belong to the same class). Splits that maximize the reduction in Gini impurity (Information Gain) are selected at each tree node.
[69]: 5.5 Gradient Boosting — Sequential Residual Fitting
[70]: The Gradient Boosting model builds an additive ensemble where each new weak learner h_m(x) is fit to the negative gradient (residuals) of the current model's loss function:
[71]: F_m(x) = F_{m-1}(x) + η · h_m(x)   η = learning rate (shrinkage)
[72]: For binary classification, the log-loss (cross-entropy) function is minimized. Each iteration progressively corrects the errors of the previous model, making Gradient Boosting particularly powerful for complex non-linear decision boundaries.
[73]: 5.6 K-Nearest Neighbors — Euclidean Distance
[74]: KNN classifies each test sample by finding the k = 5 closest training samples in feature space using Euclidean distance and assigning the majority class among those neighbours:
[75]: d(x, xᵢ) = √( Σⱼ (xⱼ − xᵢⱼ)² )   summed over all 13 features j
[76]: ŷ = majority_class({ class(x₁), ..., class(xₖ) })   k = 5 nearest neighbors
[77]: 5.7 Logistic Regression — Sigmoid Function
[78]: Logistic Regression models the probability of heart disease as a sigmoid transformation of a linear combination of the 13 input features:
[79]: P(y=1|x) = 1 / (1 + e^(−(w₀ + w₁x₁ + w₂x₂ + ... + w₁₃x₁₃)))
[80]: If P(y=1|x) ≥ 0.5, the patient is classified as at risk of heart disease. Weights w_i are optimized using L-BFGS gradient descent over up to 1,000 iterations to minimize the binary cross-entropy loss.
[81]: 5.8 Accuracy Score
[82]: Used to rank and compare all four trained models on the held-out 20% test set:
[83]: Accuracy = (True Positives + True Negatives) / Total Predictions × 100%
[84]: 5.9 F1-Score (Macro and Weighted)
[85]: F1-Score balances Precision and Recall, especially important for this dataset which exhibits significant class imbalance (70,593 disease samples vs. 160 healthy samples):
[86]: Precision = TP / (TP + FP)
[87]: Recall    = TP / (TP + FN)
[88]: F1 = 2 × (Precision × Recall) / (Precision + Recall)
[89]: Macro F1: unweighted mean across both classes — penalises poor performance on the minority class (healthy patients). Weighted F1: mean weighted by class support — reflects overall correctness dominated by the majority class.
[90]: 5.10 TF-IDF Similarity (RAG Engine)
[91]: The RAG engine converts both the user query and all medical knowledge entries into TF-IDF vectors, then ranks entries by cosine similarity to retrieve the most relevant medical context:
[92]: TF(w, d) = count(w, d) / total_words(d)
[93]: IDF(w) = log((N + 1) / (df(w) + 1)) + 1
[94]: Cosine_Sim(q, d) = (q · d) / (||q|| × ||d||)
[95]: The top-k knowledge entries with the highest cosine similarity to the query are appended to the Gemini prompt as trusted medical context, significantly reducing hallucination risk.
[96]: 6. Algorithms and Models Used
[97]: 6.1 Data Preprocessing Pipeline
[98]: Before any model training, raw patient data flows through a four-stage preprocessing pipeline implemented across feature_pipeline.py, pipeline/scaler_pipeline.py, and the preprocessing/ module:
[99]: Dataset merging: Four source CSVs (cardio_train.csv, heart_disease.csv, heart_failure.csv, sleep_health.csv) are cleaned individually (clean_cardio.py, clean_heart_disease.py, etc.) and merged into heart_combined_dataset.csv via merge_datasets.py.
[100]: Feature selection: The 13 clinically validated features listed in Table 1.2 are extracted. The target column 'num' is binarized: 0 = No Disease, all other values = 1 (Disease).
[101]: Numeric coercion: All feature columns are cast to numeric using pd.to_numeric(errors='coerce'), converting malformed string values to NaN.
[102]: Missing value imputation: NaN values are filled using column means (SimpleImputer strategy='mean') computed only on the training subset to prevent data leakage.
[103]: StandardScaler normalization: All features are zero-mean, unit-variance normalized. The fitted scaler is serialized to scaler.pkl for consistent inference at prediction time.

It achieves 99.89% test accuracy with the fastest training time (~0.05 seconds) since KNN is a lazy learner storing the training set without building an explicit model.
[111]: 6.5 Comparative Model — Logistic Regression
[112]: Logistic Regression (LogisticRegression, max_iter=1000, solver='lbfgs') fits a linear decision boundary in 13-dimensional feature space using iterative gradient descent optimization. It achieves 99.86% test accuracy and provides the most interpretable model through its learned feature weights. It is the best choice when model explainability is a priority.
[113]: 6.6 AI Engine — Google Gemini 2.5 Flash
[114]: For tasks beyond binary classification, the system invokes Google Gemini 2.5 Flash via the google-genai library (gemini_ai.py). Gemini handles three distinct tasks: (1) Top 3 cardiac condition estimation with probability percentages based on patient features; (2) Conversational cardiology Q&A augmented by RAG context; (3) Multimodal document analysis — extracting health values from uploaded PDF or image lab reports.
[115]: 6.7 RAG Engine — TF-IDF + Cosine Similarity
[116]: The RAG engine (rag_engine.py + vector_store.py + knowledge_base.py) implements a lightweight retrieval system without requiring an external vector database. Medical knowledge entries are loaded and indexed into a TF-IDF matrix at startup. At query time, cosine similarity ranks entries and the top-k most relevant are injected into the Gemini prompt. This grounds AI responses in curated, reviewed medical facts.
